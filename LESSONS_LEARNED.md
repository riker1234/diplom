# Итоги разработки: Система подбора периферии

## О проекте

Веб-приложение для подбора компьютерной периферии (мыши, клавиатуры, мониторы, наушники, микрофоны, коврики) с агрегацией цен из нескольких магазинов (Ozon, Wildberries, Ситилинк) и системой рекомендаций на основе анкетирования.

---

## 1. Парсинг данных

### Что узнали
- Каждый магазин имеет свою структуру API и защиту от парсинга. Ozon и WB требуют Selenium для обхода JavaScript-рендеринга, Ситилинк частично доступен через статический HTML.
- WB использует CDN с формульным адресом изображений: `basket-{XX}.wbbasket.ru/vol{vol}/part{part}/{sku}/images/big/1.webp`, где номер корзины зависит от диапазона артикула. Формула устаревает — иногда нужно перебирать корзины вручную.
- Ситилинк отдаёт `og:image` в HTML без JavaScript — достаточно простого HTTP-запроса через `httpx`.

### Проблемы и решения
- **Дублирование товаров**: один и тот же товар парсился повторно. Решение — проверка по уникальному SKU перед вставкой.
- **Матчинг между магазинами**: WB-товары сопоставлялись с Ozon-записями по схожести названий (`SequenceMatcher`). Без дополнительных проверок алгоритм давал ложные совпадения у товаров с одинаковыми generic-названиями ("Мышь беспроводная чёрная"). Добавили проверку бренда и ценового соотношения (не более 2.5×).
- **Рейтинг вместо бренда**: Ozon иногда возвращал рейтинг товара ("4.9") в поле бренда. Добавили фильтрацию по regex `^\d+(\.\d+)?$`.
- **Аксессуары в каталоге**: поисковый запрос "микрофон" возвращал съёмные микрофоны для гарнитур. Добавили фильтрацию по ключевым словам в названии ("съёмный", "для гарнитур").

## 2. Проектирование базы данных

### Что узнали
- Денормализованная схема (6 отдельных таблиц по категориям) оправдана когда:
  - у каждой сущности принципиально разные атрибуты
  - запросы всегда в рамках одной категории (нет межтабличных JOIN)
  - парсеры работают независимо
- Полиморфная ссылка (таблица `store_availability` с полями `product_type` + `product_id`) позволяет связать одну вспомогательную таблицу со всеми товарными без FK-ограничений.
- SKU у разных магазинов независимы — одно число может означать разные товары на Ozon и WB. Поэтому хранятся отдельные поля: `ozon_sku`, `wb_sku`, `citilink_sku`.

### Ошибки проектирования (выявлены при тестировании)
- Фильтр по цене в API изначально учитывал только `price` (Ozon). Исправлено: используется OR по всем площадкам для `price_max`, AND-условие для `price_min`.
- Поле `size` у ковриков хранится как строка ("360x300мм") — SQL-фильтрация невозможна, пришлось парсить числа в Python.

## 3. Рекомендательная система

### Архитектура
Двухэтапный подход:
1. **SQL-фильтрация** — отсеивает заведомо неподходящие товары (бюджет, тип подключения, форм-фактор)
2. **Скоринг в Python** — начисляет очки за соответствие предпочтениям, сортировка по `(-score, best_price)`

### Что узнали
- Бюджетный фильтр через `OR` по всем площадкам правильнее чем только по Ozon — пользователь может купить товар где угодно.
- Скоринг должен использовать `best_price` (минимум по всем площадкам), а не только Ozon-цену.
- Некоторые ответы ("any", "не важно") должны явно пропускаться в фильтрах — иначе возвращается пустой результат.
- Границы числовых диапазонов требуют точности: монитор 24" попадал одновременно в "маленький" (≤24") и "средний" (≥24") — исправлено на строгое неравенство.

### Выявленные баги в логике скоринга
- RGB-подсветка коврика давала +1 балл всегда, не учитывая ответ пользователя.
- Скоринг выгодной цены считался только от Ozon-цены, игнорируя WB и Ситилинк.

## 4. Frontend (React + TypeScript + Tailwind)

### Что узнали
- TypeScript-интерфейсы стираются в рантайме — нельзя экспортировать их как значения. Импорт должен быть `import type { Foo }`, иначе Vite выдаёт ошибку.
- Vite по умолчанию слушает только `localhost` (IPv6). Для доступа по IP нужен `server: { host: '0.0.0.0' }`.
- Для мультивыбора в фильтрах `Set<string>` удобнее массива: O(1) проверка наличия, чистый toggle через `set.add/delete`.
- Логика фильтра источников: **пересечение (AND)** — показывать только товары у которых есть ВСЕ выбранные источники. Это позволяет найти товары, доступные сразу на нескольких площадках для сравнения цен.

### Типичные ошибки
- `?? 0` в сортировке "сначала дороже" ставил товары без цены на первое место. Исправлено на `?? -Infinity`.
- Компонент `ImageBox` без `onError`-обработчика показывал сломанные изображения вместо заглушки.

## 5. Качество данных

### Главный урок
Данные из внешних источников всегда требуют очистки:
- Названия — generic ("Мышь беспроводная") совпадают у разных товаров
- Бренды — могут содержать рейтинги, "ОРИГИНАЛ", пустые строки
- Цены — могут отсутствовать (NULL) на одних площадках
- Изображения — URLs устаревают, CDN меняет структуру

### Инструменты очистки написанные в проекте
| Скрипт | Назначение |
|--------|-----------|
| `cleanup_wb_mismatch.py` | Отвязывает WB от товаров с подозрительной разницей цен |
| `cleanup_microphones.py` | Удаляет аксессуары из таблицы микрофонов |
| `backfill_wb_images.py` | Восстанавливает изображения WB через перебор корзин |
| `backfill_citilink_images.py` | Подтягивает изображения Ситилинк через og:image |
| `diagnostics.py` | Полная диагностика БД: дубли, нулевые цены, фейковые бренды |

## 6. Итог

| Технология | Применение |
|-----------|-----------|
| Python + FastAPI | REST API, бизнес-логика |
| SQLAlchemy + PostgreSQL | ORM, работа с БД |
| Selenium + httpx | Парсинг магазинов |
| React + TypeScript | SPA-интерфейс |
| Tailwind CSS | Стилизация |
| Vite | Сборка фронтенда |

Ключевой вывод: **реальные данные из внешних источников дают больше инженерных задач, чем сама бизнес-логика**. Большая часть работы ушла не на алгоритмы рекомендаций, а на борьбу с качеством данных, ложными совпадениями и устаревшими URL.
