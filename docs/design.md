# СППР по подбору периферии для ПК — Документ проектирования

---

## Часть 1. Исследование

### 1. Проблема

Пользователь, желающий подобрать периферию для ПК (мышь, клавиатуру, монитор, наушники и т.д.), сталкивается с перегрузкой выбором: на маркетплейсах DNS и Wildberries — тысячи позиций с непонятными техническими характеристиками. Чтобы сделать обоснованный выбор, нужно самостоятельно изучить десятки обзоров и сравнений. Это занимает часы и всё равно не гарантирует результата.

### 2. Пользователь

**Геймер-любитель, 18–28 лет**, который собирает или апгрейдит игровой стол. Разбирается в играх, но не в железе. Знает, чего хочет по ощущениям («лёгкая мышь», «клавиатура с приятным звуком»), но не знает, как это выражается в характеристиках. Имеет чёткий бюджет.

### 3. Гипотезы

- Обычный поиск на DNS/Wildberries тратит слишком много времени — фильтры есть, но без понимания характеристик ими невозможно воспользоваться.
- Люди покупают неподходящую периферию, потому что не знают, что такое сенсор, переключатели или матрица монитора.
- Рутина исследования повторяется при каждой покупке: пользователь каждый раз проходит один и тот же путь от нуля.
- Экспертный совет (форумы, Reddit) помогает, но медленный — нужно ждать ответа, описывать ситуацию.

### 4. Анализ конкурентов

| Конкурент | Какую проблему решает | Где хорош | Где ломается |
|---|---|---|---|
| **DNS / Citilink** | Покупка периферии с фильтрацией | Каталог, цены, наличие в магазинах | Нет умных рекомендаций, фильтры требуют экспертизы |
| **Wildberries / Ozon** | Покупка с отзывами | Широкий ассортимент, быстрая доставка | Нет технической фильтрации, много шума |
| **PC Part Picker** | Сборка ПК с проверкой совместимости | Отличная логика совместимости комплектующих | Периферия не охвачена совсем |
| **Reddit / 4PDA** | Экспертный совет от сообщества | Живая экспертиза, учёт контекста | Медленно, нужно описывать ситуацию, не масштабируется |
| **YouTube-обзоры** | Визуальное знакомство с продуктом | Доверие, живые впечатления | Нет персонализации, нет прямой ссылки на покупку |

### 5. Идея

> **Мы делаем веб-приложение (СППР) для геймеров-любителей, чтобы решить проблему потери времени и неудачных покупок при выборе периферии для ПК.**

Пользователь отвечает на 3–5 простых вопросов («для чего используешь», «какой бюджет», «нужен ли провод»), система возвращает отфильтрованный список подходящих устройств из реальных магазинов.

### 6. MVP

Минимально жизнеспособный продукт: **анкета → список рекомендаций**.

Если убрать:
- историю подборов — продукт работает ✓
- сравнение товаров — продукт работает ✓
- авторизацию — продукт работает ✓
- интеграцию с корзиной магазина — продукт работает ✓
- парсинг в реальном времени — продукт работает (на предзаполненной БД) ✓

Если убрать анкету или список результатов — продукт мёртв. Это и есть ядро.

### 7. Ограничения проекта

- Разрабатывается одним человеком в рамках дипломной работы.
- Ограниченное время: нет возможности реализовать ML-рекомендации, мобильное приложение, продвинутую аналитику.
- Данные о товарах заполняются вручную или через парсинг — нет официального API от магазинов.
- Нет бюджета на хостинг и инфраструктуру: развёртывание локальное.

## Часть 2. Проектирование

# 1. Анализ требований и планирование проекта

## 1.1. Описание предметной области

Система поддержки принятия решений (СППР) для подбора периферийных устройств персонального компьютера. Предметная область — рынок компьютерной периферии: мыши, клавиатуры, мониторы, наушники, микрофоны, коврики. Система помогает пользователю сформулировать требования к устройству и получить список подходящих товаров, не требуя от него технических знаний.

## 1.2. Проблема, которую решает проект

Пользователи тратят значительное время на самостоятельное изучение характеристик и обзоров при выборе периферии. Технические параметры (сенсор, тип переключателей, матрица монитора) непонятны без специальных знаний. Фильтры на маркетплейсах рассчитаны на экспертов, а не на обычных пользователей.

## 1.3. Цель проекта

Разработать веб-приложение, которое на основе простого опроса пользователя формирует персонализированный список рекомендуемых периферийных устройств из реальной базы товаров.

## 1.4. Задачи проекта

1. Собрать и хранить базу данных периферийных устройств (6 категорий).
2. Реализовать механизм опроса пользователя (анкета с понятными вопросами).
3. Реализовать алгоритм трансляции ответов анкеты в фильтры базы данных.
4. Отобразить результаты подбора в понятном интерфейсе.
5. Реализовать автоматическое обновление базы товаров из открытых источников.

## 1.5. Целевая аудитория

Пользователи ПК без технической экспертизы, преимущественно геймеры-любители 18–28 лет, планирующие покупку периферии.

## 1.6. Основной пользователь системы

**Геймер-любитель**: знает, какой результат хочет получить от устройства («быстрый отклик», «тихие нажатия»), но не знает, каким характеристикам это соответствует. Имеет ограниченный бюджет и хочет быстро получить надёжный совет.

## 1.7. Сценарии использования

**Сценарий 1 — Подбор мыши:**
1. Пользователь открывает приложение и выбирает категорию «Мышь».
2. Отвечает на вопросы: «Для чего используете?» → «Игры», «Нужна ли беспроводная?» → «Нет», «Бюджет» → «до 3000 ₽».
3. Получает список из 5–10 мышей, отсортированных по соответствию.
4. Нажимает на товар и переходит на страницу магазина.

**Сценарий 2 — Подбор монитора:**
1. Выбирает категорию «Монитор».
2. Отвечает: «Основное использование» → «Игры», «Размер» → «24–27"», «Бюджет» → «до 25 000 ₽».
3. Получает список мониторов с частотой ≥144 Гц в заданном ценовом диапазоне.

## 1.8. Состав MVP

- Каталог товаров по 6 категориям с фильтрацией.
- Анкета подбора для каждой категории (3–5 вопросов).
- Алгоритм трансляции ответов в SQL-фильтры.
- Страница результатов с карточками товаров.
- Страница товара со ссылкой на магазин.
- База данных, предзаполненная товарами (ручная/парсинг).

## 1.9. Что не входит в MVP

- Авторизация и личный кабинет.
- История и сохранение подборов.
- Сравнение товаров side-by-side.
- Покупка внутри приложения.
- Мобильное приложение.
- ML/AI-рекомендации.
- Отзывы пользователей.

## 1.10. Методология разработки

Итеративная разработка: backend-first, затем frontend. Функциональность наращивается итерациями, каждая итерация заканчивается рабочей версией.

## 1.11. Итерации проекта

| Итерация | Содержание |
|---|---|
| 1 | Модели БД, миграции, базовый FastAPI CRUD |
| 2 | Парсеры DNS и Wildberries, заполнение БД |
| 3 | Движок рекомендаций (анкета → фильтры) |
| 4 | React-фронтенд: главная, анкета, результаты |
| 5 | Страница товара, шлифовка UI, тестирование |

## 1.12. Риски и ограничения проекта

| Риск | Митигация |
|---|---|
| Изменение структуры страниц DNS/WB (парсинг сломается) | Предзаполнить БД вручную для MVP |
| Нет официального API магазинов | Использовать парсинг с осторожностью |
| Ограниченное время разработки | Чёткий MVP, откладывать всё лишнее |
| Одиночная разработка | Упрощённая архитектура без микросервисов |

# 2. Требования к программному обеспечению

## 2.1. Назначение системы

Веб-приложение для формирования персонализированных рекомендаций по выбору компьютерной периферии на основе опроса пользователя.

## 2.2. Границы системы

**Входит в систему:** хранение базы товаров, опрос пользователя, фильтрация и ранжирование, отображение результатов, ссылки на магазины.

**Не входит:** оформление заказа, оплата, гарантийное обслуживание, управление складом.

## 2.3. Функциональные требования

| ID | Требование |
|---|---|
| F-01 | Система хранит товары 6 категорий: мышь, клавиатура, монитор, наушники, микрофон, коврик |
| F-02 | Пользователь может выбрать категорию и запустить анкету |
| F-03 | Анкета содержит 3–5 понятных вопросов для каждой категории |
| F-04 | По ответам формируется отфильтрованный список товаров |
| F-05 | Список отображает название, цену, ключевые характеристики и изображение |
| F-06 | Нажатие на товар открывает ссылку на магазин |
| F-07 | Система поддерживает ручное обновление базы товаров через парсеры |
| F-08 | API возвращает список товаров с фильтрацией по параметрам |

## 2.4. Нефункциональные требования

| ID | Требование |
|---|---|
| NF-01 | Время ответа API при подборе — не более 500 мс |
| NF-02 | Интерфейс работает в современных браузерах (Chrome, Firefox, Edge) |
| NF-03 | Приложение разворачивается локально без облачной инфраструктуры |
| NF-04 | Код покрыт тестами не менее чем на 70% |

## 2.5. Требования к данным

- База содержит не менее 20 товаров в каждой категории.
- Каждый товар имеет: название, бренд, цену, технические характеристики, ссылку на магазин.
- Цены актуализируются при запуске парсера.

## 2.6. Требования к интерфейсу пользователя

- Главная страница: сетка из 6 карточек категорий.
- Страница анкеты: последовательные вопросы с вариантами ответа.
- Страница результатов: карточки товаров с кнопкой «Смотреть в магазине».
- Адаптивная вёрстка для разрешений от 768px.

## 2.7. Требования к API

- REST API на базе FastAPI.
- Эндпоинты для каждой категории с фильтрацией по характеристикам.
- Эндпоинт рекомендаций: принимает ответы анкеты, возвращает список товаров.
- Документация через Swagger UI (автогенерация FastAPI).

## 2.8. Требования к безопасности

- API не требует аутентификации (публичное MVP).
- Валидация входных данных через Pydantic.
- CORS ограничен адресом фронтенда.
- SQL-инъекции исключены ORM (SQLAlchemy).

## 2.9. Требования к производительности

- Ответ на запрос подбора: < 500 мс при базе до 1000 товаров.
- Парсер обновляет базу в фоновом режиме, не блокируя API.

## 2.10. Требования к надёжности

- При ошибке парсера существующие данные в БД сохраняются.
- При отсутствии результатов подбора система возвращает пустой список с сообщением, а не ошибку.

## 2.11. Критерии приемки

- Пользователь может пройти полный путь: выбор категории → анкета → результаты → переход в магазин.
- Все 6 категорий содержат товары.
- API-тесты проходят для всех эндпоинтов.

# 3. Архитектурное проектирование

## 3.1. Общая архитектура системы

```
┌─────────────────┐        HTTP/JSON       ┌──────────────────────┐
│  React Frontend │ ◄────────────────────► │  FastAPI Backend     │
│  (Vite + React) │                        │  (Python 3.11)       │
└─────────────────┘                        └──────────┬───────────┘
                                                      │ SQLAlchemy ORM
                                           ┌──────────▼───────────┐
                                           │  PostgreSQL 15        │
                                           │  (Docker)             │
                                           └──────────────────────┘
                                                      ▲
                                           ┌──────────┴───────────┐
                                           │  Parsers (DNS, WB)   │
                                           │  + APScheduler       │
                                           └──────────────────────┘
```

## 3.2. Архитектурный стиль

Монолитное REST-приложение с разделением на слои: роутеры → бизнес-логика → репозиторий (ORM) → БД. Фронтенд — отдельное SPA-приложение, взаимодействующее через API.

## 3.3. Основные модули системы

| Модуль | Назначение |
|---|---|
| `app/models/` | SQLAlchemy-модели таблиц БД |
| `app/schemas/` | Pydantic-схемы для валидации запросов/ответов |
| `app/routers/` | FastAPI-роутеры (эндпоинты) по категориям |
| `app/recommendation/` | Движок рекомендаций: вопросы и алгоритм фильтрации |
| `app/parsers/` | Парсеры DNS и Wildberries + APScheduler |
| `frontend/src/` | React-компоненты, страницы, API-клиент |

## 3.4. Ответственность модулей

- **models** — описание схемы данных, не содержит бизнес-логики.
- **schemas** — валидация и сериализация данных на границе API.
- **routers** — маршрутизация HTTP-запросов, вызов логики.
- **recommendation** — единственное место, где живёт логика подбора.
- **parsers** — изолированная логика сбора данных из внешних источников.

## 3.5. Взаимодействие модулей

```
HTTP Request → Router → Recommendation Engine → ORM Query → PostgreSQL
                     ↑
              Pydantic Schema (валидация входа)
                     ↓
              Pydantic Schema (сериализация ответа)
```

## 3.6. Внешние сервисы и интеграции

| Сервис | Тип интеграции | Назначение |
|---|---|---|
| DNS-shop.ru | HTTP-парсинг (BeautifulSoup) | Источник товаров и цен |
| Wildberries API | REST API (публичный) | Источник товаров и цен |

## 3.7. Технологический стек

| Слой | Технология | Версия |
|---|---|---|
| Backend framework | FastAPI | 0.115 |
| ORM | SQLAlchemy | 2.0 |
| Миграции | Alembic | 1.13 |
| Валидация | Pydantic | 2.x |
| База данных | PostgreSQL | 15 |
| DB-драйвер | psycopg[binary] | 3.x |
| Парсинг HTML | BeautifulSoup4 | 4.12 |
| Фоновые задачи | APScheduler | 3.10 |
| Frontend | React + Vite | 18 / 5 |
| HTTP-клиент (FE) | Axios | — |
| Роутинг (FE) | React Router | 6 |
| Тестирование | pytest + httpx | — |
| Контейнеризация | Docker Compose | — |

## 3.8. Обоснование выбранных технологий

- **FastAPI** — автоматическая генерация OpenAPI-документации, нативная поддержка Pydantic, высокая производительность на async-запросах.
- **SQLAlchemy 2.0** — зрелый ORM с поддержкой async, защита от SQL-инъекций.
- **PostgreSQL** — надёжная СУБД с поддержкой индексов и сложных запросов; запускается через Docker без установки.
- **React + Vite** — быстрая сборка, компонентная модель, большая экосистема.

# 4. Проектирование данных

## 4.1. Основные сущности системы

`Mouse`, `Keyboard`, `Monitor`, `Headphones`, `Microphone`, `Mousepad`, `StoreAvailability`

## 4.2. Модель данных

Все товарные сущности имеют общие поля:

| Поле | Тип | Описание |
|---|---|---|
| id | Integer PK | Первичный ключ |
| name | String | Полное название товара |
| brand | String | Производитель |
| price | Float | Цена в рублях |
| dns_product_id | String | ID товара на DNS |
| wb_sku | String | SKU товара на Wildberries |
| urls | JSON | Словарь ссылок: `{"dns": "...", "wb": "..."}` |
| updated_at | DateTime | Время последнего обновления |

## 4.5. Индексы и ограничения

- Индекс по `price` для каждой товарной таблицы (фильтрация по бюджету — самый частый запрос).
- Уникальное ограничение на `dns_product_id` и `wb_sku` для предотвращения дублей при upsert.
- `NOT NULL` на `name`, `brand`, `price`.

# 5. Проектирование бизнес-логики

## 5.1. Основные процессы системы

1. **Подбор по анкете** — пользователь отвечает на вопросы, система возвращает отфильтрованные товары.
2. **Обновление базы** — парсеры собирают товары из DNS и Wildberries, upsert в БД.
3. **Просмотр каталога** — прямая фильтрация по параметрам через API.

## 5.2. Алгоритмы работы ключевых функций

### Алгоритм рекомендации

```
Входные данные: {category: "mouse", answers: {use_case: "gaming", wireless: "no", budget: "mid"}}

1. Загрузить вопросник для категории (questions.py)
2. Для каждого ответа получить соответствующий набор фильтров:
   - use_case: "gaming" → weight_max: 100, sensor: ["PixArt", "Hero"]
   - wireless: "no" → connection_types: "USB"
   - budget: "mid" → price_min: 2000, price_max: 5000
3. Объединить фильтры в один SQL-запрос (AND-логика)
4. Выполнить запрос к БД
5. Вернуть список товаров, отсортированных по цене
```

### Алгоритм upsert при парсинге

```
Для каждого товара из парсера:
1. Найти существующую запись по dns_product_id / wb_sku
2. Если найдена → обновить price, urls, updated_at
3. Если не найдена → создать новую запись
```

## 5.3. Состояния объектов

**Товар:**
- `актуальный` — updated_at < 24 часов
- `устаревший` — updated_at ≥ 24 часов (требует обновления)

## 5.4. Правила обработки ошибок

| Ситуация | Поведение |
|---|---|
| Парсер не может достучаться до DNS | Логируем ошибку, пропускаем итерацию, данные в БД не трогаем |
| Нет товаров по запросу подбора | Возвращаем `200 OK` с пустым массивом и сообщением |
| Неверные параметры фильтра | Pydantic вернёт `422 Unprocessable Entity` |
| Товар не найден по ID | Возвращаем `404 Not Found` |

## 5.5. Правила валидации данных

- `price` > 0
- `weight_g` > 0 (для мыши)
- `refresh_rate_hz` ∈ {60, 75, 100, 144, 165, 240, 360} (для монитора)
- `diagonal_inch` ∈ [17, 49]
- `budget` в анкете ∈ допустимых значений вопросника

## 6.3. Формы ввода данных

- `.env` с паролями добавлен в `.gitignore`.

- CORS: разрешён только `http://localhost:5173`.
- Admin-эндпоинт: проверка заголовка `X-Admin-Key`.
- Rate limiting — не реализуется в MVP.

# 12. Итоговое описание решения

## 12.1. Что спроектировано

Веб-приложение «СППР по подбору периферии для ПК» — двухзвенная система с FastAPI-бэкендом и React-фронтендом, использующая PostgreSQL для хранения каталога из 6 категорий периферийных устройств (мышь, клавиатура, монитор, наушники, микрофон, коврик). Система собирает данные из DNS и Wildberries через парсеры, предоставляет REST API с фильтрацией и движком рекомендаций на основе анкеты.

## 12.2. Какие задачи закрывает система

- Устраняет необходимость разбираться в технических характеристиках вручную.
- Сокращает время выбора периферии с часов до минут.
- Предоставляет актуальные цены и ссылки на магазины.
- Даёт персонализированный список под конкретные задачи и бюджет.

## 12.3. Какие ограничения остались

- Нет авторизации и сохранения истории подборов.
- Данные зависят от стабильности парсеров (DNS/WB могут изменить структуру страниц).
- Нет ML-рекомендаций — только правиловая фильтрация.
- Развёртывание только локальное, не облачное.
- Охват только 6 категорий периферии.

## 12.4. Возможности дальнейшего развития

- Добавить авторизацию и личный кабинет с историей подборов.
- Реализовать ML-ранжирование результатов (коллаборативная фильтрация).
- Расширить категории (веб-камеры, колонки, игровые кресла).
- Добавить сравнение товаров.
- Интегрировать официальные API магазинов при наличии доступа.
- Добавить мобильное приложение на React Native.
- Реализовать облачное развёртывание (Docker + Nginx + VPS).
