# Проектный документ
# СППР по подбору периферии для ПК

---

## 1. Анализ требований и планирование проекта

### 1.1. Описание предметной области

Компьютерная периферия — категория товаров, включающая устройства ввода и вывода информации: мыши, клавиатуры, мониторы, наушники, микрофоны, коврики для мыши. Рынок этих товаров представлен сотнями производителей и тысячами SKU. Один и тот же товар продаётся на нескольких маркетплейсах по разным ценам. Технические характеристики записаны в произвольном формате и отличаются от источника к источнику.

Пользователь, выбирающий, например, механическую клавиатуру, вынужден одновременно изучать параметры (тип переключателей, форм-фактор, подключение), сравнивать цены на Ozon, Wildberries, Ситилинк и при этом понимать, что именно важно для его сценария использования.

### 1.2. Проблема, которую решает проект

Покупатель компьютерной периферии тратит от 30 до 90 минут на самостоятельный подбор одного устройства: изучает характеристики, читает отзывы, сравнивает цены на нескольких сайтах. При этом технические термины (DPI, импеданс, тип матрицы) понятны не всем, а фильтры на маркетплейсах заточены под поиск конкретной модели, а не под сценарий использования.

### 1.3. Цель проекта

Разработать систему поддержки принятия решений (СППР), которая помогает пользователю подобрать подходящую компьютерную периферию через диалог о потребностях, не требуя от него знания технических характеристик.

### 1.4. Задачи проекта

1. Реализовать сбор данных о товарах с маркетплейсов Ozon, Wildberries, Ситилинк.
2. Нормализовать и хранить характеристики товаров в единой базе данных.
3. Разработать API для фильтрации и поиска товаров.
4. Реализовать движок рекомендаций на основе анкеты пользователя.
5. Разработать веб-интерфейс для взаимодействия с системой.

### 1.5. Целевая аудитория

Люди в возрасте 16–35 лет, которые собирают или обновляют рабочее или игровое место, не являются профессиональными обзорщиками техники и хотят потратить минимум времени на выбор.

### 1.6. Основной пользователь системы

**Артём, 22 года, студент.** Впервые покупает игровую мышь. Знает, что есть "игровые" и "офисные", видел обозначения DPI и RGB, но не понимает, что выбрать при бюджете 2000 рублей. Не хочет читать длинные обзоры — хочет получить три конкретных варианта с ценами.

### 1.7. Сценарии использования

**Сценарий 1 — Быстрый подбор через анкету:**
Пользователь переходит на сайт → выбирает категорию "Мышь" → отвечает на 3–4 вопроса (бюджет, назначение, тип подключения) → получает список из 5–10 рекомендованных товаров с ценами и ссылками на магазины.

**Сценарий 2 — Ручная фильтрация:**
Пользователь знает, что хочет (например, механическую клавиатуру TKL с Cherry MX Red до 5000 руб.) → применяет фильтры → видит отфильтрованный каталог.

**Сценарий 3 — Сравнение цен:**
Пользователь уже знает модель → ищет её в каталоге → видит цены на всех доступных маркетплейсах.

### 1.8. Состав MVP

- Сбор данных с Ozon, Wildberries, Ситилинк по 6 категориям.
- REST API с фильтрацией по ключевым параметрам каждой категории.
- Анкета-рекомендатор для каждой категории (3–5 вопросов).
- Веб-интерфейс: главная страница, страница категории, страница рекомендаций.
- Отображение цен по источникам и ссылки на товар.

### 1.9. Что не входит в MVP

- Личный кабинет и история запросов.
- Отслеживание изменения цен во времени.
- Отзывы и оценки пользователей.
- Мобильное приложение.
- Автоматическое обновление данных по расписанию.

### 1.10. Методология разработки

Итеративная разработка без формального Scrum. Каждая итерация — одна категория товаров или один крупный модуль. Фиксация результатов через git-коммиты с описанием изменений.

### 1.11. Итерации проекта

| Итерация | Результат |
|---|---|
| 1 | Базовая архитектура, модели БД, парсер Ozon для мышей |
| 2 | Расширение моделей, парсеры для всех 6 категорий (Ozon) |
| 3 | Интеграция Wildberries, нормализация данных |
| 4 | Интеграция Ситилинк (Selenium), нормализация сенсоров и типов |
| 5 | REST API, схемы, движок рекомендаций |
| 6 | Веб-интерфейс (React + Vite) |

### 1.12. Риски и ограничения проекта

| Риск | Влияние | Митигация |
|---|---|---|
| Изменение структуры страниц маркетплейсов | Высокое | Модульная архитектура парсеров, быстрое обновление |
| Блокировка парсера маркетплейсом | Высокое | Selenium-stealth, случайные задержки |
| Неполные данные (null-поля) | Среднее | Fallback на название товара, несколько ключевых полей |
| Ограниченное время разработки (1 разработчик) | Высокое | Чёткий MVP, отказ от второстепенных функций |

## 2. Требования к программному обеспечению

### 2.1. Назначение системы

Система предназначена для автоматизированного подбора компьютерной периферии на основе потребностей пользователя с агрегацией данных из нескольких источников.

### 2.2. Границы системы

**Входит в систему:** сбор данных с маркетплейсов, хранение и нормализация характеристик, API, движок рекомендаций, веб-интерфейс.

**Не входит:** оформление заказа, авторизация пользователей, оплата, хранение пользовательских сессий.

### 2.3. Функциональные требования

| ID | Требование |
|---|---|
| FR-01 | Система хранит товары по 6 категориям: мыши, клавиатуры, мониторы, наушники, микрофоны, коврики |
| FR-02 | Для каждого товара хранится цена, характеристики и ссылки на магазины |
| FR-03 | API позволяет фильтровать товары по параметрам категории |
| FR-04 | Движок рекомендаций принимает ответы на анкету и возвращает ранжированный список |
| FR-05 | Интерфейс отображает рекомендации с ценами, изображениями и ссылками на магазины |
| FR-06 | Администратор может запустить парсинг через API |

### 2.4. Нефункциональные требования

- **Производительность:** ответ API не более 300 мс при базе до 2000 товаров.
- **Доступность:** система работает локально; развёртывание через Docker Compose.
- **Читаемость кода:** единый стиль, типизация через Pydantic и SQLAlchemy.

### 2.5. Требования к данным

- Каждый товар однозначно идентифицируется по SKU источника.
- Один товар может быть связан с несколькими источниками (fuzzy matching по названию).
- Числовые параметры хранятся в нормализованном виде (DPI — integer, вес — граммы).
- Строковые параметры типа "тип подключения" нормализуются к стандартным значениям.

### 2.6. Требования к интерфейсу пользователя

- Одностраничное приложение (SPA) на React.
- Выбор категории на главной странице.
- Анкета с 3–5 вопросами, навигация "назад/вперёд".
- Страница результатов: карточки товаров с ценой, изображением, характеристиками, кнопкой перехода в магазин.
- Страница каталога с фильтрами по основным параметрам.

### 2.7. Требования к API

- REST, JSON, HTTP.
- Эндпоинты для каждой категории: `GET /category/` (список с фильтрами), `GET /category/{id}` (детальная карточка).
- Эндпоинт рекомендаций: `POST /recommendation/`.
- Эндпоинты администрирования: `POST /admin/parse/{category}`.

### 2.8. Требования к безопасности

Система работает в локальной сети (учебная среда). Административные эндпоинты защищены API-ключом. Парсеры не хранят и не передают персональные данные пользователей.

```
                    │Routers │  │ Reco  │  │Parsers  │
                    │  /API  │  │Engine │  │Ozon/WB/ │
                    └─────┬──┘  └───┬───┘  │Citilink │
                          │         │      └──┬──────┘
                          └────┬────┘         │
                               │              │
                        ┌──────▼──────────────▼──┐
                        │  PostgreSQL (Docker)    │
                        │  port 5433              │
                        └────────────────────────┘
```

### 3.2. Архитектурный стиль

**Layered Architecture (слоистая архитектура):**
- **Presentation layer** — React SPA
- **API layer** — FastAPI роутеры + Pydantic схемы
- **Business logic layer** — движок рекомендаций, нормализация данных
- **Data access layer** — SQLAlchemy ORM
- **Infrastructure layer** — парсеры, PostgreSQL, Docker

### 3.3. Основные модули системы

| Модуль | Путь | Назначение |
|---|---|---|
| Parsers | `app/parsers/` | Сбор данных с маркетплейсов |
| Models | `app/models/` | ORM-модели таблиц БД |
| Schemas | `app/schemas/` | Pydantic-схемы для API |
| Routers | `app/routers/` | HTTP эндпоинты |
| Recommendation | `app/recommendation/` | Логика анкеты и скоринга |

### 3.4. Ответственность модулей

**Parsers:** каждый парсер отвечает за один источник. Реализует функции `parse_<category>(db)` и `backfill_<category>(db)`. Не знает о других источниках.

**Models:** декларативное описание таблиц. Содержат вычисляемое свойство `source` для определения происхождения записи.

**Routers:** только HTTP-слой. Принимают query-параметры, вызывают запрос к БД, возвращают схему.

**Recommendation Engine:** принимает `{category, answers}`, строит SQL-запрос с фильтрами, применяет скоринг, возвращает топ-20.

### 3.5. Взаимодействие модулей

```
Парсер → записывает данные → PostgreSQL
Роутер → читает данные → PostgreSQL → отдаёт Pydantic-схему → SPA
Рекомендатор → строит запрос → PostgreSQL → скоринг в памяти → SPA
```

### 3.6. Внешние сервисы и интеграции

| Сервис | Роль | Метод доступа |
|---|---|---|
| Ozon | Источник данных | Playwright (headless browser) + внутренний JSON API |
| Wildberries | Источник данных | Публичный CDN API (`basket-XX.wbbasket.ru`) + браузер |
| Ситилинк | Источник данных | Selenium + selenium-stealth, JS-инъекция для характеристик |

### 3.7. Технологический стек

| Уровень | Технология |
|---|---|
| Backend | Python 3.12, FastAPI 0.115 |
| ORM | SQLAlchemy 2.0 |
| База данных | PostgreSQL 15 (Docker) |
| Миграции | Alembic |
| Парсинг | Playwright, Selenium, selenium-stealth, requests |
| Frontend | React 18, TypeScript, Vite |
| Контейнеризация | Docker, Docker Compose |

### 3.8. Обоснование выбранных технологий

**FastAPI** — автоматическая генерация OpenAPI-документации, нативная типизация через Pydantic, высокая производительность.

**PostgreSQL** — поддержка `ILIKE` для нечёткого поиска строк, надёжность при конкурентной записи во время парсинга.

**Playwright/Selenium** — маркетплейсы используют клиентский JavaScript для рендеринга характеристик; без headless-браузера данные недоступны.

**React + Vite** — быстрая сборка, большая экосистема UI-компонентов, подходит для SPA с несколькими страницами.

### 4.4. Связи между таблицами

Связь "один товар — несколько источников" реализована через nullable-поля `*_sku` в рамках одной строки. При парсинге нового источника существующая запись дополняется через fuzzy matching по названию (порог схожести 0.82 по алгоритму SequenceMatcher).

### 4.5. Нормализация данных

**`connection_types`** приводится к одному из значений:
- `"проводная"` — только проводное подключение
- `"беспроводная"` — только беспроводное (Bluetooth, радиоканал)
- `"проводная/беспроводная"` — оба режима

**`keyboard_type`** приводится к: `"Механическая"`, `"Мембранная"`, `"Магнитно-механическая"`, `"Оптическая"`.

**`sensor`** — отбрасываются значения, не являющиеся моделью чипа (строки типа "беспроводная", DPI-спецификации вида "12000 dpi").

### 4.6. Индексы и ограничения

- `UNIQUE` на `ozon_sku`, `wb_sku`, `citilink_sku`, `dns_sku` — предотвращает дублирование.
- Первичные ключи `id` по всем таблицам.

## 5. Проектирование бизнес-логики

### 5.1. Основные процессы системы

**Процесс 1 — Парсинг данных:**
```
Поисковый запрос → маркетплейс → список товаров →
для каждого товара: получить характеристики →
нормализовать значения →
fuzzy match с существующими записями →
создать новую запись или дополнить существующую
```

**Процесс 2 — Рекомендация:**
```
Пользователь отвечает на анкету →
POST /recommendation/ {category, answers} →
Движок строит SQL-фильтры по обязательным параметрам →
Скоринг каждого оставшегося товара →
Сортировка: по убыванию баллов, затем по цене →
Топ-20 результатов
```

### 5.2. Алгоритм скоринга рекомендаций

Базовый балл = 0. Начисление баллов:

| Условие | Баллы |
|---|---|
| Цена ≤ 70% от бюджета | +2 |
| Мышь: вес ≤ 80г + сценарий "игры" | +3 |
| Клавиатура: тип переключателей совпал | +3 |
| Монитор: частота ≥ 144 Гц + "игры" | +3 |
| Монитор: IPS-матрица + "работа" | +2 |
| Наушники: микрофон есть + "игры" | +2 |
| Наушники: микрофон есть + "звонки" | +3 |
| Микрофон: конденсаторный + "стриминг" | +2 |

### 5.3. Алгоритм fuzzy matching при парсинге

1. Из названия нового товара удаляются служебные слова (цвет, артикул, размер).
2. Вычисляется схожесть со всеми существующими записями категории по `SequenceMatcher`.
3. Если максимальная схожесть ≥ 0.82 — запись считается совпадением; поля нового источника дописываются в существующую строку.
4. Иначе — создаётся новая запись.

### 5.4. Нормализация типа подключения

```python
has_wireless = bool(re.search(r'беспровод|bluetooth|радиоканал', low))
# Убираем "беспровод*" перед поиском корня "провод"
cleaned = re.sub(r'беспровод\w*', '', low)
has_wired = bool(re.search(r'провод', cleaned)) or (usb AND not донгл)

→ "проводная/беспроводная" | "беспроводная" | "проводная"
```

Ключевая деталь: слово "беспроводная" удаляется перед поиском корня "провод", чтобы избежать ложного детектирования проводного подключения.

### 5.5. Правила валидации данных при парсинге

- Товар не добавляется, если все `required_fields` равны NULL (фильтрация нерелевантных результатов поиска).
- Числовые поля парсятся регулярными выражениями; при неудаче сохраняется NULL.
- Булевы поля определяются по ключевым словам: "да", "есть", "yes", "+", "rgb".
- Сенсор отбрасывается, если значение соответствует паттерну DPI-спецификации или типа подключения.

## 8. Проектирование тестирования

### 8.1. Что тестируется

### 8.2. Unit-тесты (примеры)

```python
def test_normalize_connection_type():
    assert _normalize_connection_type("Bluetooth; USB Type-C") == "проводная/беспроводная"
    assert _normalize_connection_type("беспроводное") == "беспроводная"
    assert _normalize_connection_type("USB") == "проводная"

def test_sanitize_sensor():
    assert _sanitize_sensor("12000 dpi, ускорение 35G") is None
    assert _sanitize_sensor("беспроводная") is None
    assert _sanitize_sensor("PixArt PAW3395") == "PixArt PAW3395"
```

### 8.3. API-тесты (примеры)

```python
def test_list_mice_with_price_filter(client):
    r = client.get("/mice/?price_max=2000")
    assert r.status_code == 200
    for item in r.json():
        assert item["price"] <= 2000

def test_recommendation_returns_sorted_by_score(client):
    r = client.post("/recommendation/", json={
        "category": "mouse",
        "answers": {"budget": 3000, "use_case": "gaming"}
    })
    assert r.status_code == 200
    scores = [item["score"] for item in r.json()]
    assert scores == sorted(scores, reverse=True)
```

### 11.6. Логирование и мониторинг

FastAPI автоматически логирует все HTTP-запросы через Uvicorn. Парсеры выводят в stdout статистику: количество добавленных, обновлённых, пропущенных и упавших записей. При ошибке парсинга отдельного товара транзакция откатывается (`db.rollback()`), остальные товары продолжают обрабатываться.

Пример вывода парсера:
```
{'added': 12, 'updated': 38, 'failed': 0, 'skipped': 4}
```

### 11.7. Резервное копирование

Для учебного проекта резервное копирование выполняется вручную через `pg_dump`:

```bash
docker exec backend-db-1 pg_dump -U postgres diplom > backup.sql
```

Восстановление:
```bash
docker exec -i backend-db-1 psql -U postgres diplom < backup.sql
```

### 11.8. Обновление данных

Данные обновляются через административные эндпоинты API:

```bash
# Полный парсинг категории
POST /admin/parse/mice

```

Парсинг выполняется по требованию. Автоматического расписания в текущей версии нет.
